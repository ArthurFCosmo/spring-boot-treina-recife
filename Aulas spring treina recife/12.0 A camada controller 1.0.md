Agora vamos para nossas classes de controladores. Essas serão as classes responsáveis por receber e enviar requisições ao cliente.

Criamos o pacote controller para armazenar nossas classes, e criamos a classe UsuarioController.

Nela vamos anotar:

```Java
@RestController 
@RequestMapping(value = "/usuarios")
class UsuarioController {

	...

}
```

> `@RestController` - Essa anotação informa ao Spring que essa será uma classe controller.

> `@RequestMapping(value = "/URL")`- Essa anotação define um endereço para as requisições. Ex: Toda requisição do cliente que vier endereçada com `/usuario` será direcionada para essa classe.

Após isso, vamos fazer a injeção dos nossos métodos services que criamos no último pacote, daquela mesma forma instanciando um objeto da classe e anotando com `@Autowired`.

```Java
	@Autowired
	private UsuarioSerivce usuarioService;
```

## 🔷 Método para salvar um Usuário

```Java
@PostMapping  
public ResponseEntity<Usuario> cadastrar(@RequestBody Usuario usuario) {
	return ResponseEntity.status(HttpStatus.CREATED)
		.body(usuarioService.salvarUsuario(usuario));
}
```

Esse método é mais complexo quando comparado com os métodos que criamos nas outras camadas da API, vamos destrinchar por partes.

#### A anotação

> `@PostMapping` - Essa anotação mostra para o Spring que o método a seguir realizará uma operação POST no nosso CRUD, ou seja, gravar dados no nosso BD.

##### A declaração:

`public ResponseEntity<Usuario> cadastrar(@RequestBody Usuario usuario) {...}`

> `ResponseEntity<>` - Esse método genérico define qual será a Classe ou tipo da resposta, como nós fazemos normalmente no java, mas fica como parâmetro desse método. No nosso caso vamos retornar um `Usuario`, então devemos declarar dentro dos `<>`.

> `@RequestBody` - Após declarar o nome do nosso método, declaramos o parâmetro, que vai ser usuario da classe `Usuario`, um objeto. Até ai nada demais. Mas com essa anotação, informamos ao Spring que esse objeto virá no corpo da requisição.

##### O retorno

Nosso método praticamente não terá rotina, ele retornará a execução de alguns métodos diretamente.

```Java
return ResponseEntity.status(HttpStatus.Created)
	.body(usuarioservice.salvarUsuario(usuario));
```

> *Isso é apenas uma linha de código, mas quebrei em duas para facilitar a leitura.*

Na primeira parte definimos o status http que nossa API dará ao cliente em resposta à requisição, através do `ResponseEntity.status(HttpStatus.Created)`.

Após isso, definimos o corpo da resposta que será enviada para dentro da API, contendo o método de salvar o objeto (puxado da nossa classe `service`) que foi recebido no corpo da requisição: `.body(usuarioservice.salvarUsuario(usuario))`.

> *Há um jeito mais fácil de declarar essa linha de devolver o status HTTP e realizar a operação principal, aprenderemos em breve.*

## 🔷 Método para listar os usuários existentes

```Java
@GetMapping
public ResponseEntity<List <Usuario>> listar() {
    return ResponseEntity.ok(usuarioService.listarUsuarios()); 
}
```

#### A anotação

> `@GetMapping` - Essa anotação mostra para o Spring que o método a seguir realizará uma operação GET no nosso CRUD, ou seja, buscar dados no nosso BD.

#### A declaração

`public ResponseEntity<List<Usuario>> listar() {...}`

> `ResponseEntity<List<Usuario>>` - Novamente utilizando o método genérico `ResponseEntity<>` para definir o tipo de retorno que nosso método vai dar. Dessa vez, retornaremos uma lista `List<>` contendo vários (ou nenhum) objetos da classe Usuario.

Nosso método não receberá parâmetros, pois ele não precisa de um usuário específico, ou outro tipo de parâmetro, ele é limitado a fazer apenas essa ação de buscar todos os usuários.

#### O retorno

`return ResponseEntity.ok(usuarioService.listarUsuarios());`

Dessa vez utilizamos uma linha de código mais compacta para retornar ao nosso cliente o status HTTP.

> Ao invés de usar `ResponseEntity.status(Httpstatus.Created)`, usamos o `ResponseEntity.ok()`, que faz a mesma função de retornar o status HTTP para o cliente, e recebe como parâmetro o método que vai ser executado na nossa API.

> Dentro desse método de Status, colocamos `usuarioService.listarUsuarios` como parâmetro, e esse método da nossa camada de serviço será executado.

Ficou bem mais simples assim.

## 🔷 Método para buscar um usuário específico pelo Id dele

```Java
@GetMapping("/{id}")
public ResponseEntity<Usuario> buscarPeloId(@PathVariable Long id) {
    Usuario usuario = usuarioService.buscarUsuario(id); 
  
    if (Objects.isNull(usuario)) {   
        return ResponseEntity.notFound().build(); 
    }  
  
    return ResponseEntity.ok(usuario);
}
```

#### A anotação

> `@GetMapping("/{id}")` - Essa anotação complementa a anotação da nossa classe controller `@RequestMapping(value = "/usuario")`.  Significa que requisições que tiverem `/usuario/QUALQUER VALOR DE ID` serão direcionadas para esse método. Esse `{id}` funciona como uma variável.

#### A declaração

> `public ResponseEntity<Usuario>` - Assim como antes , estamos definindo a classe do objeto que será retornado.

Continuando a análise ainda na mesma linha de código:

> `buscarPeloId(@PathVariable Long id) {...}` - Temos a declaração do nome do método, e dentro dos parênteses passamos os parâmetros normalmente. Nesse método, como é uma busca por Id, obviamente passaremos o Id. Também temos uma anotação.

> `@PathVariable` - Essa anotação que colocamos antes do parâmetro fala para o Spring que aquele parâmetro será passado na URL da requisição. Ela funciona em conjunto com o nosso `@GetMapping("/{id}")`.

#### A rotina

*Essa rotina é um pouco mais complexa, pois precisamos nos preocupar com o que acontece quando o cliente pedir o id de um usuário que não existe.*

Depois de receber o Id do usuario alvo através do parâmetro que recebemos da requisição, vamos tentar buscá-lo no BD:

```Java
	Usuario usuario = usuarioService.(buscarUsuario(id));
```

Nessa linha, estamos chamando o método service de buscar pelo Id e armazenando o resultado dentro de um uma instância da classe `Usuario` chamada usuario. Estamos fazendo isso para decidir o que acontece caso esse usuário seja nulo (o id não corresponda ao de nenhum usuario existente).

---

Depois disso colocaremos a condição, usando a classe `Objects` e o método `.iNnull()` para passar um status HTTP de not found para o usuário caso o objeto seja realmente nulo.

```Java
	if (Objects.isNull(usuario)) {
		return ResponseEntity.notFound().bluid();
	}
```

> `ResponseEntity.notFound().bluid()` - Esse método da classe `ResponseEntity` serve para retornar um Not Found para o cliente, ele deve vir acompanhado desse método `.bluid()` no fim, para construir a resposta HTTP com o corpo vazio. Ele não funciona sozinho.

---

Depois disso, basta nós fazermos o "else", para caso o Id passado como parâmetro corresponda a algum usuário no BD.

```Java
	return ResponseEntity.ok(usuario);
```

## 🔷 Método para deletar um usuario

```JAVA
@DeleteMapping("/{id}")
public ResponseEntity<Void> excluir(@PathVariable Long id) {
    Usuario usuario = usuarioService.buscarUsuario(id);  
  
    if (Objects.isNull(usuario)) {  
        return ResponseEntity.notFound().build();  
    }  
  
    usuarioService.excluirUsuario(id);
  
    return ResponseEntity.noContent().build(); 
}
```

#### A anotação

> `@DeleteMapping("/{id}")` - Essa anotação indica que quando for passada uma requisição DELETE com a URL `/usuario/ID`, deve ser recebida por essa classe.

#### A declaração

> `public ResponseEntity<Void>` - Estamos definindo que o método não retornará objetos, apenas o status HTTP.

O resto da declaração é igual à de buscar pelo id, com recebimento do Id como parâmetro através da URL.

#### A rotina

A rotina inicia da mesma forma, instanciando um `Usuario` com o método de buscar pelo id (da nossa camada services)

Depois, a condição do Id da requisição não corresponder a nenhum user do BD.

Após isso, no nosso "else", executamos mais um método da nossa camada service:

```Java
	usuarioService.excluirUsuario(id);
```

Esse método exclui o Usuario do nosso BD. Ele é executado assim "solto" pois não retorna nenhum tipo de objeto para nosso cliente.

--- 

Depois, vamos retornar o status HTTP para nosso cliente.

```java
	return ResponseEntity.noContent().bluid();
```

> `ResponseEntity.noContent()` - Retorna uma requisição HTTP com corpo vazio. Asim como o Not Found, precisamos usar o método `.bluid()`.
