O `UsuarioController` da v2.0 ganhou vários endpoints novos, além de uma mudança importante na forma de receber parâmetros.

---

## Novos tipos de parâmetro: @RequestParam

Na v1.0, só usávamos `@PathVariable` (valor na URL) e `@RequestBody` (valor no corpo). Na v2.0, entra um terceiro tipo: **`@RequestParam`**.

`@RequestParam` lê os parâmetros da **query string** — aquela parte da URL que vem depois do `?`:

```
GET /usuarios/busca?nome=Arthur
GET /usuarios/filtro?dataInicio=2000-01-01&dataFinal=2005-12-31
```

---

## Buscando por nome

```Java
@GetMapping("/busca")
public ResponseEntity<List<DadosUsuario>> buscarPeloNome(@RequestParam String nome) {
    return ResponseEntity.ok(usuarioService.buscarPeloNome(nome));
}
```

A URL para chamar esse endpoint seria:
```
GET localhost:8080/usuarios/busca?nome=Arthur
```

---

## Filtrando pela data de nascimento

```Java
@GetMapping("/filtro")
public ResponseEntity<List<DadosUsuario>> filtrarPelaDataNascimento(
    @RequestParam LocalDate dataInicio,
    @RequestParam LocalDate dataFinal) {
    return ResponseEntity.ok(usuarioService.filtrarPelaDataNascimento(dataInicio, dataFinal));
}
```

A URL para chamar esse endpoint seria:
```
GET localhost:8080/usuarios/filtro?dataInicio=2000-01-01&dataFinal=2005-12-31
```

Repara que o Spring converte automaticamente a String da URL para `LocalDate`.

## Buscando por CPF e por e-mail

```Java
@GetMapping("/cpf/{cpf}")
public ResponseEntity<DadosUsuario> buscarPeloCpf(@PathVariable String cpf) {
    DadosUsuario usuario = usuarioService.buscarPeloCPF(cpf);

    if (Objects.isNull(usuario)) {
        return ResponseEntity.notFound().build();
    }

    return ResponseEntity.ok(usuario);
}

@GetMapping("/email/{email}")
public ResponseEntity<DadosUsuario> buscarPeloEmail(@PathVariable String email) {
    DadosUsuario usuario = usuarioService.buscarPeloEmail(email);

    if (Objects.isNull(usuario)) {
        return ResponseEntity.notFound().build();
    }

    return ResponseEntity.ok(usuario);
}
```

Esses usam `@PathVariable` — o valor vai direto na URL:
```
GET localhost:8080/usuarios/cpf/123.456.789-00
GET localhost:8080/usuarios/email/arthurcosmo@gmail.com
```

## Endpoint de login

```Java
@PostMapping("/login")
public ResponseEntity<DadosUsuario> buscarPeloEmaileSenha(@RequestBody DadosLogin dadosLogin) {
    DadosUsuario usuario = usuarioService.buscarPorEmaileSenha(dadosLogin.email(), dadosLogin.senha());

    if (Objects.isNull(usuario)) {
        return ResponseEntity.notFound().build();
    }

    return ResponseEntity.ok(usuario);
}
```

Aqui usamos o `DadosLogin`, nosso DTO de entrada. O cliente envia:

```JSON
{
    "email": "arthurcosmo@gmail.com",
    "senha": "1234"
}
```

Note que acessamos os campos do record com `dadosLogin.email()` e `dadosLogin.senha()` — sem o prefixo `get`, porque em records os getters são gerados sem ele.

## Todos os endpoints do UsuarioController

| Método | URL | O que faz |
|---|---|---|
| `POST` | `/usuarios` | Cria um usuário |
| `GET` | `/usuarios` | Lista todos os usuários |
| `GET` | `/usuarios/{id}` | Busca pelo ID |
| `GET` | `/usuarios/cpf/{cpf}` | Busca pelo CPF |
| `GET` | `/usuarios/email/{email}` | Busca pelo e-mail |
| `POST` | `/usuarios/login` | Autenticação por e-mail e senha |
| `GET` | `/usuarios/busca?nome=...` | Busca por parte do nome |
| `GET` | `/usuarios/filtro?dataInicio=...&dataFinal=...` | Filtra por intervalo de nascimento |
| `PUT` | `/usuarios/{id}` | Atualiza um usuário |
| `DELETE` | `/usuarios/{id}` | Remove um usuário |

---

> 💡 **Diferença entre @PathVariable e @RequestParam:**
> - `@PathVariable` — valor faz parte do caminho da URL: `/usuarios/123`
> - `@RequestParam` — valor vai na query string: `/usuarios/busca?nome=Arthur`
