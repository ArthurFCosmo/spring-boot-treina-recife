Na v2.0, o `UsuarioService` ganhou muito mais do que métodos CRUD. Agora ele:

- Converte a entidade `Usuario` para o DTO `DadosUsuario` antes de devolver ao Controller
- Calcula a idade do usuário a partir da data de nascimento
- Mascara o CPF na resposta
- Implementa buscas por CPF, e-mail, nome e intervalo de datas

---

## O método converterParaDTO

Esse é o coração da mudança. Toda vez que o service precisa devolver um usuário ao controller, ele passa pela conversão:

```Java
private DadosUsuario converterParaDTO(Usuario usuario) {
    return new DadosUsuario(
            usuario.getId(),
            usuario.getNome(),
            usuario.getEmail(),
            mascararCPF(usuario.getCpf()),
            calcularIdade(usuario.getDataNascimento()),
            usuario.getDataNascimento(),
            usuario.getStatus().toString());
}
```

Repara que:
- O **CPF** não vai cru — passa pelo `mascararCPF()`
- A **idade** é calculada na hora — não existe no banco
- O **status** é um Enum, então precisamos do `.toString()` para virar uma String

---

## Mascarando o CPF

```Java
private String mascararCPF(String cpf) {
    return cpf.substring(0, 3) + ".***.***-**";
}
```

`substring(0, 3)` pega os 3 primeiros caracteres da String. O resto é substituído por asteriscos.

Resultado: `"123.456.789-00"` → `"123.***.***-**"`

## Calculando a idade

```Java
private Integer calcularIdade(LocalDate dataNascimento) {
    LocalDate dataAtual = LocalDate.now();
    Period periodo = Period.between(dataNascimento, dataAtual);
    return periodo.getYears();
}
```

- `LocalDate.now()` — busca a data atual do sistema
- `Period.between(dataNascimento, dataAtual)` — calcula a diferença entre duas datas em anos, meses e dias
- `.getYears()` — extrai apenas os anos da diferença

Por que usar `Period` em vez de simplesmente subtrair os anos? Porque `Period` leva em conta se o aniversário deste ano já aconteceu ou não. Se hoje é março e o aniversário é em dezembro, ele conta um ano a menos — o que é o comportamento correto.

## Os novos métodos de busca

Além dos métodos CRUD da v1.0, o service ganhou:

```Java
// SELECT * FROM TB_USUARIOS WHERE cpf = ?
public DadosUsuario buscarPeloCPF(String cpf) { ... }

// SELECT * FROM TB_USUARIOS WHERE email = ?
public DadosUsuario buscarPeloEmail(String email) { ... }

// SELECT * FROM TB_USUARIOS WHERE email = ? AND senha = ?
public DadosUsuario buscarPorEmaileSenha(String email, String senha) { ... }

// SELECT * FROM TB_USUARIOS WHERE LOWER(nome) LIKE '%?%'
public List<DadosUsuario> buscarPeloNome(String nome) { ... }

// SELECT * FROM TB_USUARIOS WHERE data_nascimento BETWEEN ? AND ?
public List<DadosUsuario> filtrarPelaDataNascimento(LocalDate dataInicio, LocalDate dataFinal) { ... }
```

Cada um desses métodos chama um método correspondente no `UsuarioRepository` — que você vai ver na próxima seção.

## Todos os métodos também retornam DTO agora

Na v1.0, `salvarUsuario()` retornava `Usuario`. Na v2.0, retorna `DadosUsuario`:

```Java
// v1.0
public Usuario salvarUsuario(Usuario usuario) {
    return usuarioRepository.save(usuario);
}

// v2.0
public DadosUsuario salvarUsuario(Usuario usuario) {
    return converterParaDTO(usuarioRepository.save(usuario));
}
```

O `listarUsuarios()` também mudou — agora usa `.stream().map()` para converter cada elemento da lista:

```Java
public List<DadosUsuario> listarUsuarios() {
    return usuarioRepository.findAll()
        .stream()
        .map(usuario -> converterParaDTO(usuario))
        .toList();
}
```

`.stream()` transforma a lista em um fluxo de elementos, `.map()` aplica a conversão em cada um deles, e `.toList()` coleta o resultado de volta numa lista.
