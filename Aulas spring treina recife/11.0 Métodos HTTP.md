O frontend se comunica com nossa API por meio de requisições HTTP. Uma requisição consiste em uma URL e às vezes, pode enviar algum tipo de dado, esse dado é armazenado no `corpo da requisição`, geralmente um arquivo JSON.

Essas requisições são mapeadas na camada controller da nossa api, obedecendo o seguinte esquema: "Tal URL ativa tal operação".

### Tipos de requisição

Temos 4 tipos de método HTTP:

- GET - Buscar dados
- POST - Gravar dados
- PUT - Modificar dados já existentes
- DELETE - Deletar dados

No Spring, definimos o tipo do método com anotações, e é de extrema importância fazer essa distinção.

#### Por que é importante?

Essa distinção é importante pois cada método tem uma intenção diferente, e existe uma convenção padrão dos métodos HTTP, fazendo seu sistema ser reconhecível para outros sistemas ou outros desenvolvedores.

Só de bater o olho em um controller, você tem uma noção do que cada método faz só pelas anotações Spring atreladas a ele, mesmo sem ler a implementação.

Um exemplo disso é o uso dos métodos anotados como POST:

> Imagina que o usuário clica em "Salvar perfil" e a internet trava por 2 segundos. Ele clica de novo, achando que não funcionou.
> 
> - Se for `PUT`, não tem problema: ele está dizendo "o estado final do meu usuário é X" — repetir isso 1 ou 10 vezes dá o mesmo resultado.
> - Se fosse tudo `POST`, você criaria **dois usuários idênticos** no banco, porque POST significa "crie algo novo a cada chamada".
> 
> Isso também importa pra retry automático: navegadores, bibliotecas HTTP e até o próprio Spring sabem que é seguro reenviar um `GET` ou `PUT` que falhou por timeout — mas nunca reenviam um `POST` sozinhos, porque seria perigoso duplicar a ação.
