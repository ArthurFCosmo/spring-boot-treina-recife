## O problema de devolver a entidade diretamente

Na versão 1.0 da nossa API, os endpoints devolviam o objeto da entidade diretamente para o cliente:

```Java
public ResponseEntity<Usuario> cadastrar(@RequestBody Usuario usuario) {
    return ResponseEntity.ok(usuarioService.salvarUsuario(usuario));
}
```

Isso parece conveniente, mas cria três problemas:

- **Segurança**: o campo `senha` do usuário seria enviado na resposta para quem fez a requisição.
- **Acoplamento**: o formato da resposta fica colado à estrutura do banco. Se você mudar o nome de uma coluna, a resposta da API muda junto.
- **Rigidez**: você não consegue facilmente adicionar campos calculados na resposta (como `idade`), porque eles não existem na tabela.

A solução é criar uma camada intermediária — o **DTO**.

---

## O que é um DTO?

**DTO** significa **Data Transfer Object** (Objeto de Transferência de Dados). É uma classe que representa exatamente *o que você quer enviar para o cliente*, nem mais, nem menos.

Pensa assim: você trabalha numa empresa e vai fazer uma apresentação. Você não leva todos os seus arquivos internos para a sala de reunião — você prepara um **relatório** com apenas as informações relevantes para quem vai ver. O DTO é esse relatório.

```
Entidade (BD)         DTO (Resposta para o cliente)
─────────────         ────────────────────────────
id           ──────►  id
nome         ──────►  nome
cpf          ──────►  cpf (mascarado: "123.***.***-**")
email        ──────►  email
senha        ✗         (não vai para o cliente)
dataNascimento ──────► dataNascimento (formatada)
status       ──────►  status
             +         idade (calculada, não existe no BD)
```

## O tipo `record` do Java

Para criar DTOs, usamos o tipo `record`, disponível a partir do Java 16. Ele é perfeito para esse uso porque:

- Cria automaticamente construtor, getters e `equals/hashCode`
- É **imutável** — uma vez criado, não pode ser alterado
- Deixa o código muito mais enxuto

Comparando uma classe normal com um `record`:

```Java
// Classe normal — precisaria de construtor, getters, etc.
public class DadosUsuario {
    private Long id;
    private String nome;
    // ... construtor, getters...
}

// Record — tudo isso numa linha
public record DadosUsuario(Long id, String nome, String email, ...) {}
```

Em um `record`, os campos são acessados como métodos sem o prefixo `get`:

```Java
dadosUsuario.nome()   // e não dadosUsuario.getNome()
dadosUsuario.email()  // e não dadosUsuario.getEmail()
```

---

> 💡 **Resumo:** DTO é uma classe intermediária que define exatamente o que vai ser enviado ao cliente, protegendo dados sensíveis e permitindo transformações como mascaramento de CPF e cálculo de idade.
